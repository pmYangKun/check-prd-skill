# PRD与系统设计检查 Skill

> **使用须知：本 Skill 需要 Opus 模型才能进行深度分析。** 在 Claude Code 中运行时，请确保使用 `/model claude-opus-4-6` 切换到 Opus，或在调用时手动指定 Opus 模型。使用 Sonnet 或 Haiku 时分析深度会大幅下降。

## 用途说明

本Skill用于对B端产品的PRD（产品需求文档）和系统设计方案进行全面、系统化的质量检查。

基于三份权威参考文献构建检查体系：
- **《决胜B端（第二版）》**：B端产品设计方法论，涵盖业务调研、产品定位、架构设计、数据建模、流程设计、产品化、项目管理等全流程
- **《决胜体验设计 v1.5》**：B端用户体验设计方法论，涵盖体验四层模型、设计价值观、交互设计原则、AI功能设计六脉神剑等
- **《决胜B端PRD模板 v2.0》**：标准化PRD文档模板，定义了从项目背景到运营计划的完整文档结构

适用场景：
- PRD评审前的自查
- 设计方案质量回顾
- 新人PRD写作指导
- 产品方案完整性验证

## 使用方法

1. 准备待检查的PRD文档：提供文件路径，或直接粘贴PRD内容
2. 调用本Skill：使用 `/check-prd` 命令
3. 系统将依次从14个维度对PRD进行检查，每个维度检查完立即输出详细分析，最终输出汇总报告

## 检查执行流程

### 第零步：产品定型（必须先执行）

在开始任何维度检查前，先阅读PRD并判断以下两个维度，记录结论供后续检查使用：

**A. 商业属性**
- **自研内部系统**：企业内部使用，不对外销售，如内部OA、ERP、CRM
- **商业化产品**：对外销售或运营，存在付费客户或商业变现，如SaaS、平台产品

**B. 功能类型**（参考《决胜B端》1.3节四大分类）
- **业务型管理软件**：支撑核心业务流程运转，有复杂的业务对象和状态管理，如订单管理系统、合同管理
- **工具型软件**：提供独立功能提升效率，无复杂业务对象关系，如报表工具、文档协作
- **交易型平台**：撮合多方完成交易，如供应链平台、招采平台
- **基础服务型**：提供底层能力供其他系统调用，如权限中心、消息中心、支付系统

**根据定型结论，以下维度按条件适用：**

| 维度 | 适用条件 | 不适用时处理方式 |
|------|---------|----------------|
| 07-数据建模（ER图） | **仅适用于业务型管理软件和交易型平台** | 工具型/基础服务型：标注"不适用"，仅检查关键数据结构 |
| 10-商业分析 | **仅适用于商业化产品** | 自研内部系统：仅执行10.3节投入产出分析，其余小节跳过 |
| 竞品分析（03维度内） | **仅适用于商业化产品** | 自研内部系统：替换为"同类系统参考"即可 |
| 06-架构（企业架构层6.5-6.9） | **仅适用于0-1新系统设计或系统级规划PRD** | 迭代需求/模块级需求：6.5-6.9标注"不适用"并跳过，只做6.1-6.4 |
| R8-多租户（风险项） | **仅适用于商业化SaaS产品** | 非SaaS：标注"不适用" |

记录定型结论后，继续以下检查。

### 检查输出规范（强制执行，不得违反）

**核心原则：逐维度即时输出。**

每读取一个维度文件并完成检查后，必须**立即输出该维度的详细分析**，然后再继续下一个维度。**禁止将所有维度的分析攒到最后统一输出**——这会导致因上下文压力每个维度只能获得浮于表面的处理。

**每个维度的输出格式（强制）：**

```
## 维度[编号] - [名称] ｜ 评级：[优秀 / 合格 / 待改进 / 严重缺失]

### 具体发现

**发现 1：[问题标题]** [P0/P1/P2/P3]
- PRD定位：第X节/[功能名称]
- 问题描述：[具体说明为何存在问题，不要泛泛而谈]
- 改进示例：[可以立即执行的具体方案，不是方向性建议]

**发现 2：...**
...

### 隐性问题推断
结合此PRD的产品类型、业务场景，列出PRD没写但按道理必须考虑的问题。
```

**最低输出要求：**
- 每个维度至少输出 **3条具体发现**（若确实无问题，须逐条说明理由，不得仅写"无问题"或"该维度表现良好"）
- 每条发现必须引用 PRD 中具体的章节号、功能名、流程名，让作者能一眼定位
- 每条发现必须给出具体可执行的改进示例，而非"建议补充"或"建议改进"这类虚化表述

**维度09（交互设计）的额外要求：**
- 必须从 PRD 中提取所有具体界面、弹窗、表单、操作，逐一做组件级评估
- 使用 09-ux 文件 9.6节规定的格式逐一输出每个组件的分析结论
- 若 PRD 对某界面描述不详细，也要明确指出"第X节未说明XX的交互方式，需补充"

## 汇总报告要求

完成所有维度的详细分析输出后，基于已有分析生成以下汇总报告（此报告是索引，不是替代，详细内容在各维度输出中）：

### 1. 产品定型说明
说明本PRD的商业属性（自研/商业化）和功能类型（业务型/工具型/交易型/基础服务型），以及据此哪些维度不适用或调整了检查标准。

### 2. 各维度发现摘要
| 维度 | 适用性 | 评级 | 关键发现（一句话） |
|------|--------|------|-----------------|
| 01-业务分析 | 适用 | - | |
| 02-产品类型 | 适用 | - | |
| 03-产品定位 | 适用 | - | |
| 04-场景分析 | 适用 | - | |
| 05-文档结构 | 适用 | - | |
| 06-架构设计（含企业架构层） | 适用/部分适用 | - | |
| 07-数据建模 | 适用/不适用 | - | |
| 08-流程角色 | 适用 | - | |
| 09-交互设计 | 适用 | - | |
| 10-商业分析 | 适用/不适用 | - | |
| 11-MVP策略 | 适用 | - | |
| 14-运营方案与效果跟踪 | 适用 | - | |
| 12-异常处理与健壮性 | 适用 | - | |
| 13-AI功能 | 适用/不适用 | - | |

### 3. 重大风险项（如有）
列出触发的重大风险项（R1-R8），这些是产品设计中需要优先关注的底线问题，建议在推进开发前解决。

### 4. 问题清单（全量汇总，按P0-P3排序）
从各维度详细分析中汇总所有问题，按优先级排序：
- **P0-高优先级**：影响核心业务流程或存在合规风险，建议尽快处理
- **P1-建议修订**：重要遗漏，建议上线前解决
- **P2-可改进项**：有提升空间，按资源情况安排
- **P3-优化建议**：锦上添花，有时间再做

### 5. 亮点记录
从各维度详细分析中汇总PRD做得好的地方。

### 6. Top 10改进建议
按优先级列出最重要的10条改进建议，每条包含：具体问题（引用PRD位置）、改进方向（可执行）、参考依据。

## 注意事项

- 检查的目的是帮助PRD作者发现盲点、提供改进方向，而非判定优劣
- 不同产品形态、不同阶段的PRD，检查侧重点本就不同，请结合产品定型结论灵活判断
- 局部模块PRD：商业分析、产品定位等宏观维度可参考上位文档，不强求独立完整
- 迭代需求PRD：MVP策略、系统架构等维度可适当调整
- AI功能维度仅在PRD涉及AI功能时评估，否则标记为"不适用"
